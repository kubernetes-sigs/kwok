# Optimizing etcd performance

Some applications rely on the etcd component for distributed key-value storage. Since etcd is present in both KWOK 
and non-KWOK clusters, various scenarios can be simulated to test etcd performance at scale.
[Kyverno](https://github.com/kyverno/kyverno), for example, is a practice of this.

The need for performance testing arose when Kyverno users faced issues with excessive admission reports generated by admission 
controllers that overwhelmed the etcd database which stores these configuration data. 
To predict the size a production cluster may consume, they advised users to conduct performance evaluation using KWOK in their cluster.
This setup involves creating fake virtual nodes with KWOK to host workloads and policies. Kyverno, along with other monitoring
tools like the metrics-server, are also installed in the cluster to evaluate the impact of the policies on etcd storage.

This demo below provides a step-by-step guide on how to deploy Kyverno in a KIND cluster, install KWOK into it, and lastly,
conduct a performance test on etcd.

## Prerequisites

- Install [Kind](https://kind.sigs.k8s.io/docs/user/quick-start/#installation) for creating the Kubernetes clusters.
- Install [Helm](https://helm.sh/docs/intro/install/) for installing Kyverno into the cluster.
- Install [kubectl](https://kubernetes.io/docs/tasks/tools/install-kubectl-linux/)

## Create a Kubernetes Cluster

```bash
kind create cluster
```

This command sets up a local Kubernetes cluster that will serve as the environment for deploying Kyverno.

## Clone the Kyverno repository and deploy KWOK in the cluster

Navigate to the [cloned](is a policy engine that benefits from such testing.) Kyverno repository and execute the KWOK deployment script.

Before executing the below command, open the `kwok.sh` file and comment this line `export KUBECONFIG=/etc/rancher/k3s/k3s.yaml`.

```bash
./docs/perf-testing/kwok.sh
```

This script installs KWOK, which simulates Kubernetes nodes and pods to aid in the performance test.

Other scripts below, should also run inside the cloned repository.

## Create KWOK nodes

Before executing the below command, open the `node.sh` file and comment this line `export KUBECONFIG=/etc/rancher/k3s/k3s.yaml`.

```bash
./docs/perf-testing/node.sh
```

This command sets up the nodes that will host your workloads.

When prompted, enter how many nodes you need.
1 node was used for this scenario.

## Setup monitoring components

To monitor the performance, set up the necessary monitoring components.

```bash
make dev-lab-metrics-server dev-lab-prometheus
```

## Install the etcd client

```bash
kubectl apply -f https://gist.githubusercontent.com/aojea/6f17c07251cf9251f66a611d80c6a6f1/raw/c00a517341528862fe16b2fb8c40126d5dfd17b3/etcdclient.yml
```

This command makes it possible to interact with the etcd component.

## Install Kyverno

Add the Kyverno Helm repository and install Kyverno with the following commands:

```bash
helm repo add kyverno https://kyverno.github.io/kyverno/
helm repo update
helm upgrade --install kyverno kyverno/kyverno -n kyverno \
  --create-namespace \
  --set admissionController.serviceMonitor.enabled=true \
  --set admissionController.replicas=3 \
  --set reportsController.serviceMonitor.enabled=true \
  --set reportsController.resources.limits.memory=10Gi \
  --set "features.omitEvents.eventTypes={PolicyApplied,PolicySkipped,PolicyViolation,PolicyError}" \
  # --devel \
  # --set features.admissionReports.enabled=false \
```

This command installs Kyverno with specific configurations for the admission controller and reports controller.

## Deploy Kyverno pod security standards (PSS) policies

```bash
helm upgrade --install kyverno kyverno/kyverno-policies --set=podSecurityStandard=restricted --set=background=true --set=validationFailureAction=Audit --devel
```

This step installs a set of recommended security policies for running pods in a Kubernetes cluster.

## Test the Kyverno reports controller

To test the Kyverno reports controller, workloads need to be created.

Before executing the below command, open the `deployment.sh` file and comment this line `export KUBECONFIG=/etc/rancher/k3s/k3s.yaml`.

```bash
./docs/perf-testing/deployment.sh
```

When prompted, enter the following values:
Deployment count: `1`
Deployment replicas: `1`
Deployment namespace: `default`

This numbers and namespace name may vary based on your use-case.

## Assess storage footprint

To assess the storage footprint of Kyverno policies, copy the `size.sh` script into etcd and run it.
This will provide insights into the total size of policy reports stored in etcd.

```bash
kubectl cp -n default ./docs/perf-testing/size.sh etcdclient:/
kubectl exec -it etcdclient -- sh
sh size.sh
```

When prompted, enter:
`wgpolicyk8s.io/policyreports`

A similar final value should be outputted:

`The total size for wgpolicyk8s.io/policyreports is 1209166 bytes.`

Estimated to 1.2 Megabytes.

This is the size the policy consumed in etcd during this deployment.
